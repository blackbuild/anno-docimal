# AnnoDocimal 1.0 release plan

This plan records only confirmed release gates. It does not authorize further GitHub mutations, published-API changes,
or build implementation beyond the separately approved curation already recorded.

The applied issue dispositions, work items, and dependency order are maintained in
`docs/implementation/1.0-issue-curation-plan.md`.

One 1.0 tracking issue coordinates independently executable gate issues; the mandatory API grilling remains a distinct
human-decision predecessor (ADR 0050).

The tracker and every hard gate belong to a dedicated GitHub `1.0` milestone; later work does not (ADR 0051).

## Confirmed gates

- Apply the issue #37 API-grilling decisions: the per-artifact public-type classification and allowlists are recorded,
  issue #18's naming concern resolves to `AstDocumentation.extractExact`, and implementation must generate the first
  mechanical source/binary compatibility snapshots from the human allowlist rather than all public bytecode (ADRs 0026,
  0036, 0053-0055, and 0058).
- Retain the documentation authoring/template language completed through AnnoDocimal #51 and KlumAST #489: first-class
  prose/code blocks, deliberate composition, named single-pass templates, first-class links, and contextual failures.
  Apply ADR 0058's Java-shape corrections before locking the transformation-author baseline (ADRs 0056-0058).
- Design and deliver the curated transformation-author facade and documentation value model from that grilling, mapping
  known consumers and providing deliberate shims or migration for existing helpers without pulling issue #30's
  structured carrier schema into 1.0 (ADR 0047).
- Resolve issue #35 and verify the reusable source-projection task's invalidation, stale-output, up-to-date, build-cache,
  configuration-cache, and supported-Gradle-range behavior with TestKit (ADR 0028).
- Retain the stable module names and owning global-AST provider adapter delivered for issue #36 by merged PR #50; do not
  reopen KlumAST #455's separate multi-Groovy decision (ADR 0029).
- Resolve issue #32 with an AnnoDocimal-native inherited-generics regression fixture and compile the projected Java
  source to prove that supported generic API shapes remain valid (ADR 0031).
- Resolve issue #33 by normalizing annotation-member ordering and verifying identical deterministic projections across
  the supported Groovy 3, 4, and 5 lanes (ADR 0032).
- Resolve issue #9 by defining and verifying the semantic mapping from a Groovy property's documentation to its property,
  backing field, and generated accessors without projection-time JavaBeans-name inference (ADR 0033).
- Re-scope issue #19 from its Groovy-2-era checklist and complete runtime GroovyDoc interoperability, duplicate avoidance,
  generator recognition, and deterministic carrier precedence across Groovy 3, 4, and 5 (ADR 0035).
- Complete repository-owned README/docs, Javadocs, release notes, compatibility reference, supported-API documentation,
  projection semantics, and 0.x-to-1.0 migration guidance for all artifacts and the supported base/Groovy plugin pair
  (ADR 0040).
- Prefer a clean cut from provisional 0.x source APIs: migrate KlumAST, document the path, and retain shims only when
  independently useful and cheap. Do not apply this rule to separately confirmed persisted-protocol compatibility
  (ADR 0049).
- Audit all six publications and both plugin markers through a clean local-repository consumer smoke test, covering POM
  and applicable Gradle metadata, dependencies, signing, sources/Javadocs, shading, descriptors, and resolution; correct
  the generator shadow publication's current metadata/documentation-artifact gaps (ADR 0041).
- Expose identifiable CI evidence for Java 17, Groovy 3/4/5, the selected minimum and current Gradle versions,
  configuration-cache reuse, and publication/consumer smoke tests (ADR 0042).
- Document and rehearse the 1.0 release procedure for version/tag derivation, signed Maven Central staging, Plugin Portal
  publication, changelog/GitHub Release synchronization, verification, and failure recovery; automation is optional
  (ADR 0043).
- Create and complete a source-projection policy work item defining supported selection of visibility, nested
  declarations, synthetic/Groovy artifacts, signature closure, and legacy `AnnoDocGenerator` migration. Top-level class-
  file selection stays in the calling build/task layer (ADR 0044).
- Design and deliver the narrow implementation-neutral source-projection service for source text and managed output
  locations, consuming the explicit policy and excluding generator implementation types from the supported API. Decide
  the pre-1.0 `AnnoDocGenerator` migration separately rather than assuming shims (ADR 0048).
- Create and complete a projection-contract fixture suite that asserts deterministic output and recompiles projected
  Java for representative Java/Groovy declarations, generics, nested types, annotations, supported records/enums, and
  documentation carriers across Groovy 3/4/5; include issue #32 as one regression (ADR 0045).
- Create and complete shared capture-conformance fixtures proving equivalent observable normalized documentation from
  Java APT, local Groovy capture, and global Groovy capture, including declaration keys, nested declarations, parameters,
  empty docs, service discovery, and Groovy 3/4/5 (ADR 0046).

## Cross-project sequencing

- AnnoDocimal 1.0 is a hard release prerequisite for KlumAST 4.0. KlumAST development may use pre-release versions, but
  4.0 must not release against provisional AnnoDocimal 0.x APIs. No KlumAST issue or release plan is mutated from this
  repository session (ADR 0030).

## Confirmed non-gates

- Issue #30 does not block 1.0. Structured annotations target a future 2.0 release with a separate ecosystem migration
  plan (ADR 0027).
- Issue #10 does not block 1.0. Exact extraction must be named unambiguously in 1.0; inherited and `{@inheritDoc}`
  resolution follows as a distinct additive capability (ADR 0034).
- Issue #22 is closed as obsolete under the Groovy 3+ compatibility floor (ADR 0037).
- Issue #23 remains a post-1.0 enhancement pending a concrete executable-generator contract; it is not a release gate
  (ADR 0038).
- Issue #25 is closed as solution-first; a future behavioral issue may demonstrate concrete value, while generator
  technology remains internal (ADR 0039).
- Historical changelog backfill does not block 1.0; any backfill remains evidence-based rather than inferred from commit
  subjects (ADR 0040).
