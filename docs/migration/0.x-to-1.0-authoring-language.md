# Documentation-authoring migration from 0.x to 1.0

AnnoDocimal 1.0 replaces the 0.x authoring helpers rather than preserving their names as compatibility API. Persisted
`@AnnoDoc(String)` content remains readable; this migration concerns transformation-author source code.

| 0.x concept | 1.0 replacement |
|---|---|
| `DocText` | `Documentation.parse(text)` and immutable value accessors |
| `DocBuilder` / `JavadocDocBuilder` | `Documentation.builder()` and `build().render()` |
| `TemplateHandler` | explicit `Documentation.Builder.template` / `templateValues` and `Documentation.render` |
| `AnnoDocUtil.addDocumentation` | `AstDocumentation.attachText` or `AstDocumentation.attach` |
| `ASTExtractor.extractDocumentation` / `extractDocText` | `AstDocumentation.extractExact` |
| `JavaDocUtil.toLinkString` | `AstDocumentation.referenceTo` plus `Documentation.Link` |

Use `summary`, `paragraph`, and `codeBlock` to author ordered content. Use `append`, `replace`, and
`filterParameters` when composing a captured document for a generated member. Rendered parameter tags follow the final
method signature automatically during `AstDocumentation.attach`.

Absence is explicit in the 1.0 Java API. `getSummary()`, `getReturnDescription()`, and `Link.getLabel()` return
`Optional<String>`. Public arguments reject `null`; clear singular content with `clearSummary()` or `clearReturn()`,
replace categories with empty collections, and remove attached documentation with `Documentation.empty()`. Use
`paragraph` and `returns`; the redundant `appendParagraph` and `replaceReturn` aliases are not supported 1.0 methods.

The supported facade and model are JSpecify `@NullMarked`. Compatible Java and Groovy tooling can therefore interpret
unannotated API types as non-null, while the nullable comparison input of value-based `equals(Object)` is explicit.
JSpecify does not turn optional scalar accessors into nullable returns and does not replace the runtime rejection above.
KlumAST should consume this metadata directly rather than adding a second nullability-annotation standard.

A generated-declaration rewrite can stay entirely on the supported surface:

```java
Documentation captured = AstDocumentation.extractExact(source).orElse(Documentation.empty());
Documentation generated = captured.toBuilder()
        .summary("Creates {{kind}} documentation.")
        .template("kind", "builder")
        .paragraph("The generated builder preserves model semantics.")
        .codeBlock("owner.copyBuilder(source)")
        .filterParameters(List.of("source"))
        .see(AstDocumentation.referenceTo(source))
        .build();
AstDocumentation.attach(target, generated);
```

`attach` also filters parameter descriptions against the target's final signature. Template validation completes before
the carrier is replaced, so a malformed or incomplete template does not partially rewrite existing documentation.

Replace implicit template defaults and value-presence conditionals with required `{{key}}` placeholders. Supply values
explicitly as `String`s through `Map<String, String>`. `{{param:key?fragment}}` is the only conditional form; it omits the fragment only when the
generated target lacks that parameter. Do not rely on `@template` tags to supply values.

KlumAST's 4.0 migration is owned by KlumAST #461. Its authoring portion maps to this document; it does not retain any
of the 0.x AnnoDocimal helper names. The separate multi-Groovy decision remains KlumAST #455.
