# Supported-API migration from 0.x to 1.0

AnnoDocimal 1.0 deliberately replaces accidental public helper surfaces instead of carrying compatibility shims. The
persisted `@AnnoDoc(String)` and documentation-properties protocols remain readable; the clean cut applies to source APIs
and the ambiguous opinionated plugin ID.

## Nullability migration

Supported 1.0 Java types are non-null by default through JSpecify `@NullMarked`. Normal dependency metadata exposes
`org.jspecify:jspecify`; consumers that assemble classpaths or module paths manually must include that dependency as well.
Do not translate this contract into JetBrains nullability annotations or mix standards on the supported facade.

The annotations make the existing contract visible to compatible Java tooling and to Groovy tooling that interprets
JSpecify. They do not change runtime behavior: prohibited null inputs still fail, absence remains `Optional`, empty
collections and explicit clear methods retain their category-clearing semantics, and template values remain non-null
strings. Dynamic Groovy callers must continue to follow those runtime rules even when their compiler does not perform
nullness analysis.

KlumAST can consume the supported annotations directly while migrating to the 1.0 facade; it does not need nullable
adapter methods or a parallel JetBrains-annotation layer. Test-only `@Language("groovy")` annotations remain valid IDE
language-injection metadata and are unrelated to API nullability.

## Transformation-author migration

| 0.x usage | 1.0 replacement |
|---|---|
| `ASTExtractor.extractDocumentation` or `extractDocText` | `AstDocumentation.extractExact` and immutable `Documentation` |
| `AnnoDocUtil.addDocumentation(node, text)` | `AstDocumentation.attachText(node, text)` |
| `AnnoDocUtil.addDocumentation(node, builder)` | `AstDocumentation.attach(node, builder.build())` |
| `DocText` parsing, inspection, or tag replacement | `Documentation.parse`, value accessors, and `toBuilder()` |
| `DocBuilder` / `JavadocDocBuilder` | `Documentation.Builder` and `Documentation.render()` |
| `JavaDocUtil.toLinkString` | `AstDocumentation.referenceTo` |
| `InlineJavadocs.JAVADOC_PROPERTIES_SUFFIX` | No supported Java constant. Rely on the documented `__annodoc.properties` protocol resource and its class, method, and field key semantics instead. |
| Direct metadata constants, source extractors, visitors, or Groovy-version handlers | No replacement; consume the facade's supported behavior |

`extractExact` never performs inherited lookup. Do not replace an old fallback overload with implicit hierarchy traversal;
issue #10 will add separately named resolved extraction after 1.0.

The final authoring-language migration is documented in
[the focused authoring guide](0.x-to-1.0-authoring-language.md). It uses first-class prose/code blocks and explicit named
templates; generic block kinds and paragraph qualifiers remain additive options after 1.0. Current `TemplateHandler` and
specialized builder vocabulary are not compatibility bridges.

Scalar absence is explicit: `getSummary()`, `getReturnDescription()`, and `Link.getLabel()` return `Optional<String>`.
Builder and facade arguments reject `null`; use `clearSummary()`, `clearReturn()`, empty replacement collections, or
`Documentation.empty()` for deliberate removal. `paragraph` and `returns` replace the redundant 0.x-style aliases
`appendParagraph` and `replaceReturn` before the 1.0 baseline is locked. Template maps are `Map<String, String>`.

## Source-projection migration

Replace:

```java
AnnoDocGenerator.generate(classFile, outputDirectory);
```

with an explicitly policy-configured projector:

```java
SourceProjector projector = new SourceProjector(ProjectionPolicy.documentation());
Path sourceFile = projector.projectToDirectory(classFile, outputDirectory);
```

Use `projectToText` when the caller owns persistence. Collection scanning, top-level selection, and stale-output cleanup
are not implicit singleton behavior of `SourceProjector`; use `SourceProjectionTask` when a Gradle build needs managed
collection output.

`AnnoDocGenerator` is removed without a shim. `ProjectionPolicy.documentation()` is the explicit replacement for its
implicit declaration filtering: it selects public/protected declarations and named nested types, excludes synthetic and
Groovy runtime scaffolding, retains visible language-level generated APIs, and always applies signature closure. See the
[source-projection guide](../source-projection.md) before selecting broader visibility or enabling synthetic/runtime
artifacts.

The directory operation still writes one UTF-8/LF package-relative source file. Builds that project collections can now
use `SourceProjectionTask`, which owns a complete managed output tree and removes stale projections after a successful
replacement.

## Gradle migration

The 1.0 Gradle clean cut replaces `CreateClassStubs` with `SourceProjectionTask` without a shim and replaces
`com.blackbuild.annodocimal.plugin` with `com.blackbuild.annodocimal.groovy-plugin` without a plugin-ID alias. Apply
`com.blackbuild.annodocimal.base-plugin` for the neutral Java-model Javadoc integration, or apply the new Groovy plugin
when the build should also apply Gradle's Groovy plugin and compiler metadata convention. Both use the conventional
`createClassStubs` task name for the Javadoc convenience instance.

For an IDE-only mirror, register `SourceProjectionTask` with class directories, output directory, and any include or
exclude patterns owned by the consuming build. Its complete managed output is cacheable, removes stale files, and is
safe with Gradle's configuration cache. IDE model wiring remains consumer-owned; do not compile, package, or publish
the mirror as a second API.

## KlumAST consumer map

KlumAST issue #461 owns the 4.0 migration. Its current direct calls map as follows:

- `ASTExtractor` to `AstDocumentation.extractExact` and `Documentation`;
- `AnnoDocUtil` string and builder attachment to `attachText` and `attach`;
- `DocText`, `DocBuilder`, and `JavadocDocBuilder` to `Documentation` and its builder;
- `JavaDocUtil` to `referenceTo`;
- `AnnoDocGenerator` to `SourceProjector` or a configured `SourceProjectionTask` for collection output;
- application of `com.blackbuild.annodocimal.groovy-plugin` instead of the removed plugin ID; and
- `AnnoDoc` and `InlineJavadocs` remain protocol annotations.

The authoring-language contract and consumer requirements were completed jointly through KlumAST #489 and AnnoDocimal
#51. KlumAST issue #461 owns applying the complete 0.x-to-1.0 consumer migration. KlumAST #455 remains the separate
owner of the cross-project multi-Groovy build decision.

## Module-path migration

The stable automatic module names delivered by issue #36 are unchanged by this API migration. Named consumers should also
apply [the 0.x-to-1.0 module-path migration](0.x-to-1.0-module-path.md).
